# 🔒 Constraint

> *Most constraints are soft. Most people treat them as hard. The gap is where the work is.*

---

## What this category is

Constraint tools work with limits — testing whether they're real, using them as creative drivers, finding paths around them, and stripping scope down to what's actually necessary. The central insight: most stated constraints dissolve under examination. They're assumptions, habits, or organizational inertia dressed as facts.

When a constraint is genuinely fixed, these tools find what's possible within it. When it's soft, they find what it would take to move it. Either way, they produce more options than the default stance of accepting limits at face value.

---

## When to reach for it

- A project is blocked and someone says "we can't do X"
- Scope keeps growing; the original requirement is buried under what's wanted
- You're looking for creative solutions and the constraint feels like the ceiling on all thinking
- You want to try something but a rule or policy stands in the way — and you're not sure it should
- You have a hard constraint and need to find every viable path within it
- You want to know if a limitation is actually fixed before investing in working around it

---

## When *not* to reach for it

**Don't use constraint tools to rationalize ignoring real limits.** Hardness testing is about discovering genuine constraints, not dismissing them. Some constraints are real and should shape the approach.

**Don't invert constraints when the fundamental requirement is the problem.** If what's being asked for is wrong, scope reduction and constraint inversion won't fix it — that's a [decision](../decision/) or [identity](../identity/) question.

The nearest neighbor: [Logic](../logic/) — specifically [constraint-mapping](../logic/constraint-mapping.md), which maps the full constraint landscape for a decision. Use logic/constraint-mapping to understand the constraint topology; use these tools to act on it.
